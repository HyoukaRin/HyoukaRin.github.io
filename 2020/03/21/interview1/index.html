这是一篇伪面经

前些日子跟giria谈笑风生，说自己面试绝对不准备，兵来将挡，水来土掩。结果今天跟一般通过DD（下称PDD，也即Passing DD）交流之后，感觉自己十有八九已经凉了，为了尽量不重蹈阿里笔试的覆辙，我决定稍微准备一下。

C/C++

想来我的面试官应该不至于问我python相关的东西，今天跟PDD交流后，发现C++还有不少东西可能会被问到，所以在这篇blog里面回忆一下，想到哪算哪

智能指针

这个应该是c++11的新特性，我印象中有三个（两年前看的cpppp），但是现在只记得两个：unique-ptr与shared-ptr，unique-ptr表示独享指针，一旦被使用，该地址不可被其它地址共享，shared-ptr表示共享指针，大家可以一起用，只有当最后一个指针消亡后，指针所指向地址才会消亡

这些东西有些数据库里面X/S锁的味道，当时看这些玩意的时候挺不爽的，总是感觉这些指针违背了C++的编程理念，（总感觉C++的变化有点往python那边走的味道，C++17里面好像有更多类似的语言特性），而且最让人讨厌的是本来指针只要写个声明，现在却要多打这么多字！

类/继承/模板

PDD说他被问到了一个尤其神秘的问题：如何让一个类无法被继承。这个问题也太无厘头了，正确答案是，让构造函数私有。然而私有的构造函数真的有意义吗

虚方法，虚基类，虚函数表与虚函数指针，这些大家回忆回忆就行了

构造函数不能虚，析构函数可以是虚函数

模板就更头疼了，还有模板类下的模板类如何声明，嵌套的模板之类的，然而这些东西据说问的不多，大概HR也实在记不得了

左值与右值引用

这个其实蛮好玩的，大家有空可以去试试看这段代码

class test {
public:
    int operator[](int& x) {
        return 2;
    }
    int operator[](int&& x) {
        return 3;
    }
};

int main(void) {
    int x = 3;
    test t;
    cout << t[x]<<endl;
    cout << t[3]<<endl;        
    return 0;
}

值得注意的是，很多编译器会自动进行右值引用的优化，关于右值引用，这cpppp里面的话来总结最为妥当：”加入右值引用并不是为了让你真正使用它，而是为了让你方便地使用那些利用右值引用这项语言特性的代码库”。

Rendering

我已经想好了，要是HR问我rendering，我就跟他说，rendering只是兴趣爱好，我是做CV的

AA

最先是SSAA，然后是MSAA，在前向渲染中，最常用的还是MSAA，毕竟比起SSAA，MSAA并不用实际地增加片段着色器的调用次数，况且SSAA并不能减少颜色缓冲的大小

FXAA主要用于延迟渲染，尽管MSAA也可能可以用于延时渲染，但是其复杂的设置总是令程序员在延时渲染中不愿意使用它，于是，不如直接把AA当做延时渲染的一部分吧，通过梯度检查边缘，然后插值好像也不赖嘛，也不用管那些讨厌的采样，不也挺好吗？

走样与反走样

我真的忘了那一块怎么做的了，印象中是使用各种各样的卷积核，先变换到一个域然后再变回来什么的，主要是从来没用过

Shader工作流程，正向与延迟渲染

其实正向和延迟渲染在原理上并没有什么不同，对于一个三角形来说，正向和延迟渲染其实运算流程是一样的，都是对像素插值，然后确定颜色，延迟渲染的唯一好处在于每个片段只用调用一次片段着色器，在片段着色器花里胡哨的今天，这个一可以统一接口，二可以减少调用，即便是正向渲染，其部分效果，如shadow map，也需要延迟的流程进行后处理

AO/Shadow

AO和Shadow总容易让人弄混，AO其实削弱的是环境光照，在削弱环境光照的过程中，会给人一种shadow的感觉，但其实其并不是真正的shadow。SM与RSM后来还成为one-bounce全局光照的经典算法，今天面试完之后，我应该会专门写一些通过延时渲染完成全局光照的blog

法向/法向贴图/Mipmap

这些其实还好啦，只要记得把法向转到切平面就行了，mipmap本质是一个插值算法，本身其实并不会在复杂度上优化渲染流程，但是通过插值的方法，使得远处的纹理更加真实

采样/频域

我早就忘得差不多了，频域最近有个CV的论文，说在频域卷积，然后训练resnet50，要是HR真问起来我就扯这个，顾左右而言他

ACK

Thanks for the useful discussion with PDD who successfully reminds me of how ignorant I am!
